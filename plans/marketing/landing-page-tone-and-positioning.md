# Flux Landing Page: Tone & Positioning

## Status
Active design direction (v0 public-facing messaging)

---

## Objective

Define how Flux presents itself to new users on the landing page.

The goal is not to explain the system in depth.

The goal is:

> Get a developer to think: “This looks simple. I’ll try it.”

---

## Target User

Primary:
- Developers familiar with PostgreSQL
- Indie hackers / hobbyists
- Small teams prototyping products

Secondary:
- Engineers evaluating backend infrastructure for a team
- Developers frustrated with overly abstracted tools

---

## Core User Mindset

A new visitor is thinking:

```txt
I want to use PostgreSQL.
I don’t want to deal with infrastructure.
I want something simple and predictable.
I don’t want magic.
````

They are NOT thinking:

```txt
How does pooled vs dedicated architecture work?
What is the migration path?
How does schema isolation function internally?
```

---

## Core Positioning

Flux is:

```txt
PostgreSQL, ready to use.
```

Expanded:

```txt
A clean API over PostgreSQL,
without setup, without unnecessary abstraction.
```

---

## What Flux is NOT (on the landing page)

Do NOT position Flux as:

* “A full backend platform”
* “A Supabase competitor”
* “A complex infrastructure system”
* “A feature-rich toolkit”

These may be true later, but they do not help initial adoption.

---

## Messaging Layers

### 1. Hero (first 3 seconds)

Must be immediately understandable.

Examples:

```txt
PostgreSQL, ready to use.
```

```txt
Build on PostgreSQL.
Flux handles the setup.
```

---

### 2. Supporting Line

Adds clarity without complexity.

```txt
A clean API over your database.
No setup. No abstraction.
```

---

### 3. Lifecycle (secondary message)

Introduce differentiation without overwhelming:

```txt
Start free.
Upgrade to a dedicated instance when your app grows.
No rewrites.
```

Important:
This is NOT the headline.
This is reinforcement.

---

## Tone Guidelines

### 1. Simple

Avoid:

* long sentences
* nested explanations
* multiple clauses

Prefer:

```txt
Short. Direct. Clear.
```

---

### 2. Minimal

If a section can be removed without losing clarity → remove it.

The landing page should feel:

```txt
light
fast
obvious
```

---

### 3. Low Jargon

Avoid:

* “multi-tenant”
* “schema isolation”
* “JWT bridge”
* “pooled vs dedicated”

These belong in docs, not the landing page.

---

### 4. No Over-Explanation

Do not preemptively explain edge cases.

Trust the user to explore further.

---

### 5. Confident (but not hype)

Avoid:

```txt
“Revolutionary”
“Next-generation”
“Best-in-class”
```

Prefer:

```txt
matter-of-fact confidence
```

---

## Visual Philosophy

The design should match the tone:

```txt
clean
neutral
editorial
```

Avoid:

```txt
loud SaaS gradients
busy dashboards in hero
feature overload
```

Think:

```txt
calm
controlled
intentional
```

---

## Content Structure (high-level)

Landing page should follow:

1. Hero (simple statement)
2. Short supporting explanation
3. Minimal “why Flux” bullets
4. Optional visual (app → API → DB)
5. Lifecycle mention (upgrade path)
6. Call to action

---

## What to Avoid

Do NOT:

* lead with architecture
* explain internal system design
* list too many features
* compare against competitors
* overload with examples

---

## Core One-Liner

This should always feel true:

```txt
Flux is PostgreSQL without the mess.
```

---

## Secondary Framing (internal use)

Flux’s real value:

```txt
complex under the hood
simple on the surface
```

The landing page must reflect:

```txt
simple on the surface
```

---

## Acceptance Criteria

The landing page is successful if:

* A developer understands it in under 5 seconds
* It feels simple, not technical
* It encourages trying Flux, not studying it
* It does not overwhelm with detail

---

## Future Expansion

Detailed explanations belong in:

* `/docs`
* dashboard onboarding
* technical guides

Not on the landing page.

---

## Final Principle

```txt
Clarity over completeness.
Simplicity over explanation.
```
